  The Core Convention Problem

  The deepest issue is the chi¬∑Cm ambiguity. There are two valid ways to write
  the monodomain equation:

  Convention A ‚ÄĒ ŌÉ-based (physical units):
  Ōá¬∑Cm ¬∑ ‚ąāV/‚ąāt = ‚ąá¬∑(ŌÉ¬∑‚ąáV) ‚ąí Ōá¬∑I_ion
  Operator form: A = ŌáCm/dt ¬∑ M ‚ąí őł ¬∑ K(ŌÉ)

  Convention B ‚ÄĒ D-based (normalized):
  ‚ąāV/‚ąāt = ‚ąá¬∑(D¬∑‚ąáV) ‚ąí I_ion/Cm
  where D = ŌÉ/(Ōá¬∑Cm). Operator form: A = 1/dt ¬∑ I ‚ąí őł ¬∑ L(D)

  Convention A stores ŌÉ in the stiffness matrix and Ōá¬∑Cm in the mass matrix.
  Convention B absorbs Ōá¬∑Cm into D and uses identity mass. They're 
  mathematically equivalent but operationally dangerous to mix. V5.4's FDM puts
  D in L but Ōá¬∑Cm in the mass ‚ÄĒ this works only if D = ŌÉ/(Ōá¬∑Cm) and you
  consistently divide the ionic source by Cm. The bidomain engine copies this
  but the chi=1400 scaling makes the diffusion-to-mass ratio ~10‚ĀĽ‚Ā∂, meaning
  diffusion is negligible compared to the identity. The system becomes stiff in
  the wrong direction.

  LBM already uses Convention B: D goes into ŌĄ, source is R = 
  -(I_ion+I_stim)/Cm, no chi anywhere. This is simpler and less error-prone.

  Recommendation: Convention B universally. The user provides ŌÉ, Ōá, Cm as
  physical parameters. The engine converts to D = ŌÉ/(Ōá¬∑Cm) at construction time
  and uses D everywhere internally with chi_num=1, Cm_num=1.

  Would Unification Be Beneficial?

  Yes, but with important caveats about scope and timing.

  What should be unified (the "shared foundation"):

  1. Ionic models ‚ÄĒ One copy. This is the strongest argument. Bug fixes, new
  models, and LUT optimizations propagate everywhere.
  2. Mesh ‚ÄĒ One StructuredGrid with BoundarySpec. Already essentially identical
  between V5.4 and bidomain. LBM currently uses raw tensors but can trivially
  wrap the grid.
  3. Conductivity ‚ÄĒ One class that handles both monodomain and bidomain:
  class Conductivity:
      def __init__(self, sigma_i, sigma_e=None, chi=1400, Cm=1.0):
          self.D_i = sigma_i / (chi * Cm)
          self.D_e = sigma_e / (chi * Cm) if sigma_e else None
          self.D_eff = self.D_i * self.D_e / (self.D_i + self.D_e) if self.D_e
  else self.D_i
  3. User thinks in ŌÉ (what they measure), engine works in D (what it needs).
  4. Stimulus ‚ÄĒ One StimulusProtocol. LBM's mask-based approach can be a special
   case.
  5. Convention ‚ÄĒ D-based everywhere. chi_num=1, Cm_num=1 in all operators.
  6. State interface ‚ÄĒ Common base with .V_flat, .V_grid, .ionic_states, .t.
  Bidomain adds .phi_e. LBM adds .f (distribution functions).

  What should stay separate (the "paradigm frontends"):

  1. LBM internals ‚ÄĒ Lattice definitions, collision operators, streaming,
  bounce-back are unique to LBM. No shared abstraction with FDM.
  2. Classical FDM internals ‚ÄĒ Sparse Laplacian assembly, PCG, CN/BDF time
  stepping.
  3. Bidomain extensions ‚ÄĒ L_i/L_e operators, decoupled solver, null space
  handling.

  Trying to create a shared abstraction over "collision+streaming" and
  "assemble+solve" would be a leaky abstraction that helps nobody.

  Unified User API

  The payoff is a clean, consistent user-facing API:

  from cardiac_sim import StructuredGrid, Conductivity, StimulusProtocol
  from cardiac_sim import MonodomainSimulation, BidomainSimulation,
  LBMSimulation
  from cardiac_sim.tissue.stimulus import left_edge

  # Describe the problem ONCE
  grid = StructuredGrid(Nx=150, Ny=40, Lx=3.725, Ly=0.975)
  cond = Conductivity(sigma_i=1.74, sigma_e=6.25, chi=1400, Cm=1.0)
  stim = StimulusProtocol()
  stim.add_stimulus(left_edge(0.125), start=1.0, duration=2.0, amplitude=-80.0)

  # Pick the method ‚ÄĒ everything else is consistent
  sim_fdm  = MonodomainSimulation(grid, cond, stim, dt=0.01, method='fdm')
  sim_lbm5 = LBMSimulation(grid, cond, stim, dt=0.01, lattice='d2q5')
  sim_lbm9 = LBMSimulation(grid, cond, stim, dt=0.01, lattice='d2q9')
  sim_bi   = BidomainSimulation(grid, cond, stim, dt=0.01, bc='bath_coupled')

  # All return the same interface
  for state in sim_fdm.run(t_end=40.0, save_every=0.5):
      V = state.V_grid    # (Nx, Ny) ‚ÄĒ always available
      t = state.t          # float (ms)

  1. Phase 6 cross-validation will empirically reveal every friction point

  different maturity levels risks breaking the stable one.

  Proposed timeline:
  Now:           Run Phase 6A-6C with current separate engines
                 (cv_shared.py papers over the differences)

  After Phase 6: Assess friction points from cross-validation experience
                 Write unified architecture doc (like improvement.md)

  Then:          Build cardiac_sim V2 as the unified engine

  to build the highway before or after you know where the traffic goes. Phase 6
  tells you where the traffic goes.
