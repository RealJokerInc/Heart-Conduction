Chapter 16

Implementation Roadmap for Engine V5.4

This chapter maps the mathematical framework from Chapters 12–15 onto the concrete software architecture of Engine V5.4, identifying what can be reused, what must be extended, and what must be built new.

16.1   Architecture Extensions

Engine V5.4 is built around four abstract interfaces: SpatialDiscretization, DiffusionSolver, LinearSolver, and IonicSolver. The bidomain extension requires changes primarily in the first three:

16.2   Recommended Solver Stack

Based on the analysis in Chapters 13–15, the recommended solver configuration for a production bidomain engine is:

Time Integration: Strang splitting (reaction–diffusion–reaction)

Reaction Solver: Rush–Larsen (reuse from V5.4)

Diffusion Time Stepper: SDIRK2 (L-stable, second-order)

Outer Linear Solver: FGMRES(30) with tolerance $10^{-6}$

Preconditioner: Block LDU with Schur complement approximation

Sub-block Solver: AMG V-cycle (AMGX for GPU, Hypre for CPU)

Null-Space Handling: Pin one node of $\phi_e$ to zero

Expected Performance: 10–30 FGMRES iterations per time step; 10–20× slower than monodomain

16.3   Performance Expectations

The bidomain model is inherently more expensive than monodomain. Realistic expectations:

For a typical 3D cardiac mesh with $N = 10^6$ nodes, the monodomain solve takes approximately 1–5 seconds per heartbeat on a modern GPU. The bidomain solve would take approximately 10–100 seconds per heartbeat — still tractable for research and many clinical applications.
